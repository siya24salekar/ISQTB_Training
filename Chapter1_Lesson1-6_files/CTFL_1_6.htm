What is testing?

To someone who knows nothing or very little about software testing; the term ‘software testing’ has the perception of being simply the activity of executing test cases. While partially true, test execution is only one of many tasks that a typical software tester is required to do in their day-to-day work. Typically, there are testing related tasks to be performed before and after any kind of test execution.

In order to answer the question ‘what is testing?’ we first need to consider ‘why are we testing?’ as the testing can be entirely different depending on what we are trying to achieve. The following represent some simple examples to highlight some (not all) typical reasons for testing:

Objective 1 - Finding Defects

Development testing, which encompasses component, integration and system testing typically has the sole objective to find as many failures as possible. This is so that software defects are first, identified and subsequently fixed.

Objective 2 – Gaining Confidence

Acceptance testing will typically have the main objective may to ensure that the system works as expected. This is gauged by testing against requirements. By confirming that the requirements have been met, confidence in the software can be gained.

Objective 3 – Preventing Defects

By designing tests early on in the development life-cycle, some key advantages can be gained. One advantage is that by considering how something is going to be tested while it’s still in the design stage (given an appropriate audience) can influence the design itself; thus preventing errors from being developed when the design is implemented in software code. Reviewing the requirements is also an important activity which can be highly fruitful as errors / poor design can be detected before any design or software has been created.

 

The difference between testing and debugging

There is a clear difference between testing and debugging. Essentially, ‘testing’ can be considered an activity that will show failures that have been caused by defects. Whereas, ‘debugging’ a development activity that highlights the cause of a defect and subsequently fixes the code and verifies that the defect has been fixed correctly. After the debugging activity, a tester will typically confirm that the fix does actually resolve the failure, which is referred to as confirmation testing.

Make sure you understand the following keywords associated with this lesson: 

Debugging: The process of finding, analyzing and removing the causes of failures in software.

Requirement: A condition or capability needed by a user to solve a problem or achieve an objective that must be met or possessed by a system or system component to satisfy a contract, standard, specification, or other formally imposed document.

Review: An evaluation of a product or project status to ascertain discrepancies from planned results and to recommend improvements. Examples include management review, informal review, technical review, inspection, and walkthrough.

Test case: A set of input values, execution preconditions, expected results and execution postconditions, developed for a particular objective or test condition, such as to exercise a particular program path or to verify compliance with a specific requirement.

Test objective: A reason or purpose for designing and executing a test.
