Testing principles overview

The following testing principles are suggestions based on many years of the existence of professional software testing. They are not based on any standard, but merely lessons learned by many software testers over the years. Most professional software testers will be aware of these principles, but may implement them a different way than a different way than described. So, view them as a high-level list of general software testing guidelines that if used, can be adapted to suit a typical software testing process.

To help explain the testing principles, we will use the following simple software application:

The application allows a user to input an employee number along with the first letter of their surname. After clicking the ‘Submit’ button, the resulting database record is displayed.

 

Principle 1 – Testing shows presence of defects

Software testing can only show that defects exist in the software, it can never prove that defects do not exist. It can be said that software testing can reduce the likelihood of undiscovered defects existing in the software. But, if software testing results in no defects being found, then this does not prove that no defects exist in the software.

For example, a tester is assigned to task of testing our simple application example. The tester completes his planned test cases, some defects were found and subsequently fixed. The tester is now happy that all of his planned tests have been executed, and therefore the testing has finished. So there are no more defects in the software right? Well, we don’t know for sure, but we can identify some key information to give us an idea.

Unless exhaustive testing was used, not every combination would have been tested. By looking at the test cases (planned & executed), we can ascertain how much many combinations were tested which provides us with a level of confidence in the software, and gives us an idea to the risk of how many defects may still exist in the software.

If our tester found no defects, then we cannot say that there are no defects in the software. Assuming exhaustive testing was not carried out, we can say that the chances of there being any defects is lower than when before the testing was carried out. As we haven’t tested every possible combination, then we cannot state there are no defects.

Make sure you understand the following keywords associated with this lesson: 

Defect: A flaw in a component or system that can cause the component or system to fail to perform its required function, e.g. an incorrect statement or data definition. A defect, if encountered during execution, may cause a failure of the component or system.

Exhaustive testing:

A test approach in which the test suite comprises all combinations of input values and preconditions.
