Lesson 2-4

Component Testing

The aim of component testing is to find defects in and verify the functionality of separately testable software components. These components could be for example; classes, objects, programs or modules etc. The testing is typically performed in isolation from the rest of the system/software under test. Often a software component relies on inputs from other components, so in order to test a component in isolation we need some way of simulating the other related components. Stubs, drivers and simulators are used to achieve this. A common misconception about component testing, is that because it is focused on a specific area, that the testing that can be performed is very limited. This is not the case, as not only functional testing can be performed, but also non-functional testing as a resource behavior, structural testing and robustness testing.

When considering test case design for component testing, work products including the component specification can be used, and also a data model (if it exists) or any software design documentation.

In order to effectively perform component testing, access to the code will be required. In addition to this, some working programming knowledge or experience will often be required. This is why this kind of testing is traditionally performed by a Developer. The Developer will probably be the same person who wrote the code, and although this goes against the ‘Testing Independence’ philosophy discussed in another lesson, this is standard practice in most software developments. The reason for this is because component testing is normally carried out early on in the software development lifecycle, and is something done typically before the software is handed over to the test team.

 The Developer will often use the component test to act as a sanity test to conform that they are satisfied with their work before handing it over to the testers. To assist with the testing a Developer may use a unit testing framework or debugging tools. The Developer will also often fix any defects found ‘on-the-fly’ without formally documenting it.

 Often, it may be more efficient to automate the component tests. This can actually be done before the code is written and is called a ‘test-driven’ approach or ‘test-first’ approach. This method uses an iterative approach where cycles of developed test cases are developed followed by building and integrating the code. Subsequent tests are performed until a satisfactory pass rate is achieved.

   






What is required to effectively perform component testing?

All unit tests in a passed state
Access to design documentation
Access to the code
A black box
 		  

Make sure you understand the following keywords associated with this lesson: 

Stub:

A skeletal or special-purpose implementation of a software component, used to develop or test a component that calls or is otherwise dependent on it. It replaces a called

component.

Driver:

A software component or test tool that replaces a component that takes care of the control and/or the calling of a component or system.

Component testing:

The testing of individual software components.
