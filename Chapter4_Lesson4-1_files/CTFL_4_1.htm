Lesson 4-1

The Test Development Process

For the purposes of this training course, the test development process detailed within this chapter can be executed in several different ways, from a very informal way, with little or no documentation, to very formal, which is described below. The different levels of formality will be applied depending on the context of the testing involved. The context will be affected by items such as the maturity of testing and development processes, time constraints, safety or regulatory requirements, and of course the people involved.

Test Analysis:
During any kind of test analysis, the test basis documentation should be analyzed to determine what is going to be tested (essentially to identify the test conditions). A test condition can be thought of as an item or event that can be verified by testing. An example of this would be a function, transaction, quality characteristic or structural element.

Traceability:
Having the ability to establish traceability from test conditions back to the specifications and requirements allows effective impact analysis when requirements change, and determining requirements coverage for a set of tests. During test analysis the specific test approach is implemented to select the best test design techniques to use based on, among other considerations, the identified risks.

Test Design
During a test design phase, the test cases and also the test data are created and specified. A standard test case should essentially consist of a set of input values, execution preconditions, expected results and execution post conditions, which should be developed to cover specific test objectives or test conditions. There is a standard document which is often referred to by testers called the ‘Standard for Software Test Documentation’ (IEEE STD 829-1998). this document describes the content of test design specifications and test case specifications.

Expected Results:
Expected results should be created as part of the specification of a test case and include outputs, changes to data and states, and any other consequences of the test. If the expected results have not been defined, then when the test is executed an incorrect result may be interpreted as the correct one. Expected results should ideally be defined well before any test execution has taken place.

Test Implementation:
During test implementation the test cases are typically created, implemented, prioritized and placed in the test procedure specification (IEEE STD 829-1998). The test procedure should ideally specify the order of actions for the execution of a specific test. Sometimes the tests may be ran by using a test execution tool, in this case, the sequence of actions is specified in a test script, which can be thought of as an automated test procedure. The test procedures and automated test scripts are then produced into a test execution schedule, which specifies the order in which the various test procedures, and sometimes automated test scripts, are executed. A test execution schedule should consider such factors as regression tests, technical and logical dependencies and prioritization.

Make sure you understand the following keywords associated with this lesson: 

Test case specification:

A document specifying a set of test cases (objective, inputs, test actions, expected results, and execution preconditions) for a test item.

Test design:

The process of transforming general testing objectives into tangible test conditions and test cases.

Test script:

Commonly used to refer to a test procedure specification, especially an automated one.

Traceability:

The ability to identify related items in documentation and software, such as requirements with associated tests.
