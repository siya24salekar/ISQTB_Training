Lesson 4-5

Decision Table Testing

The use of decision tables has been wide-spread in the field of software testing for decades now, and are a great way to capture system requirements that contain basic logical conditions. They are often suitable for recording complex business rules that a system is to implement, as this task is difficult to organize and the decision table format lends itself nicely to this task as it displays the logical decisions in an easy to read format.

In order to create a decision table, the specification is first analyzed, and then any conditions and actions of the system are highlighted. Each input condition and action are typically stated in such a way that they must either be true or false, which is referred to as 'Boolean'.

The decision table will also contain the triggering conditions, which can be true or false, or combinations of the two for all input conditions, and the resulting actions for each combination of conditions. The layout of each column of the decision table corresponds to a business rule that defines a unique combination of conditions and which result in the execution of the actions associated with that rule. Typically the way a decision table is used is to have least one test per column in the table, which typically involves covering all combinations of triggering conditions.

The big advantage of a decision table is that it creates combinations of conditions that otherwise may not get tested. It may be applied to any situation where the action of the software depends on multiple logical decisions.

A simple example of how a decision table works:

Rules   Is it a weekday? y y y y n n n n
    Is today a public holiday? n n y y y y n n
                     
Conditions   Is it raining? y n y n y n y n
                     
Actions   Go to work x x            
    Go for a walk       x   x   x
    Watch a movie on TV     x   x   x

Make sure you understand the following keywords associated with this lesson: 

Decision table:

A table showing combinations of inputs and/or stimuli (causes) with their associated outputs and/or actions (effects), which can be used to design test cases.

Decision table testing:

A black box test design technique in which test cases are designed to execute the combinations of inputs and/or stimuli (causes) shown in a decision table.
