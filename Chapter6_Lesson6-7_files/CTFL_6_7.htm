Lesson 6-7

Potential Benefits and Risks of Tool Support for Testing (for all tools)

Its an easy mistake to think that by simply buying or leasing a test tool, you will guarantee success. There are many types of tools, and many tools that essentially do the same job. But its important to consider not just the tool itself, but clearly define exactly what you want the tool to achieve. Due to the time, cost and effort of implementing a tool, care should be taken to ensure it will not only perform the required purpose now, but well into the future. There are many potential benefits and opportunities with using tools in relation to testing, but there are also many risks.


Potential benefits of using tools:

  • Greater consistency and repeatability (e.g., tests executed by a tool in the same order with the
    same frequency, and tests derived from requirements)

  • Repetitive work is reduced (e.g., running regression tests, re-entering the same test data, and
    checking against coding standards)

  • Ease of access to information about tests or testing (e.g., statistics and graphs about test
    progress, incident rates and performance)

  • Objective assessment (e.g., static measures, coverage)


Risks of using tools:

  • Unrealistic expectations for the tool (including functionality and ease of use)

  • Underestimating the time and effort needed to achieve significant and continuing benefits from
    the tool (including the need for changes in the testing process and continuous improvement of
    the way the tool is used)

  • Underestimating the time, cost and effort for the initial introduction of a tool (including training
    and external expertise)

  • Underestimating the effort required to maintain the test assets generated by the tool

  • Risk of tool vendor going out of business, retiring the tool, or selling the tool to a different vendor

  • Poor response from vendor for support, upgrades, and defect fixes

  • Risk of suspension of open-source / free tool project

  • Unforeseen, such as the inability to support a new platform

  • Over-reliance on the tool (replacement for test design or use of automated testing where manual testing would be better)

  • Neglecting version control of test assets within the tool

  • Neglecting relationships and interoperability issues between critical tools, such as requirements management tools, version control tools, incident management tools, defect tracking tools and tools from multiple vendors

Make sure you understand the following keywords associated with this lesson: 

Test tool:

A software product that supports one or more test activities, such as planning and control, specification, building initial files and data, test execution and test analysis.
