Lesson 6-8

Special Considerations for Some Types of Tool

Test Execution Tools

Test execution tools work by executing test objects via automated test scripts. In order to get the best out of this type of tool, significant effort is required. While at first, simply capturing tests by recording the actions of a manual tester seems great, but this method does not lend itself when using large numbers of automated test scripts. The captured script itself is a linear representation with specific data and actions that form part of each script. This type of script does work, although in a limited way, and when an error occurs, it may very well become unstable.

A popular technique used among testers today is using a data-driven testing approach that separates out all of the test inputs (which is the data). This is usually placed into a spreadsheet. Then a generic test script can read the input data and execute the same test script with different data. Often, inexperienced testers who are not familiar with scripting languages can create the test data within the spreadsheet for these predefined scripts.

Other techniques that are used in data-driven techniques are where instead of hard-coded data combinations placed that are placed in a spreadsheet, data is generated using algorithms based on configurable parameters at run time and supplied to the application directly. An example of this where a tool may use an algorithm, which generates a random user-ID, and for repeatability in pattern, a seed is employed for controlling randomness.

When using a keyword-driven testing approach, the spreadsheet will include keywords that describe the actions to
be taken (action words), and test data. Testers can then define tests using the keywords, which can be modified to suit to the application being tested. This method is again useful for inexperienced testers that do not have specific scripting skills.

Some level of technical expertise in the scripting language is required for all approaches that have just been mentioned. While inexperienced testers can use them effectively, skill will be required to create and maintain them. Whichever scripting technique is used, the expected results for each test need to be stored for later comparison.

Static Analysis Tools

If used effectively static analysis tools when applied to source code can help to enforce coding standards. But sometimes, if static analysis tools are applied to existing code, they may generate a large amount of warning messages. The warning messages themselves will not stop the code from being translated into an executable program, but in order to be effective, they should ideally should be addressed by responsible personnel to ensure that maintenance of the code is easier in the future. A good approach is to gradually introduce the analysis tool with initial filters setup to exclude some messages, an is an effective approach to ensure the users are not flooded with warning messages which can be overwhelming.

Test Management Tools

Test management tools work by interfacing with other tools (or even spreadsheets) in order to produce meaningful information in a format that fits the needs of the intended user or organization in general.

Make sure you understand the following keywords associated with this lesson: 

Data driven testing:

A scripting technique that stores test input and expected results in a table or spreadsheet, so that a single control script can execute all of the tests in the table. Data driven testing is often used to support the application of test execution tools such as capture/playback tools.

Keyword driven testing:

A scripting technique that uses data files to contain not only test data and expected results, but also keywords related to the application being tested. The keywords are interpreted by special supporting scripts that are called by the control script for the test.
